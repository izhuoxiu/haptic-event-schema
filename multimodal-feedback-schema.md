# 1. 项目背景

## 1.1 人类感官的缺失：为什么需要标准触觉语言

触觉是人类最早发育、最直接的身体感知通道，但在数字世界中，它长期被忽视。我们通过屏幕“看”到内容，通过扬声器“听”到声音，却极少通过震动“感受”到信息。

随着手机线性马达、穿戴设备、VR/AR 控制器、智能座舱、触觉手套等硬件普及，触觉反馈的硬件基础已经成熟。然而，**软件层面缺乏一套统一的描述语言**——每个平台（iOS、Android、Windows）各有自己的 API，开发者不得不为每个硬件重复编写波形代码，内容创作者也无法像撰写字幕一样“撰写触觉”。

此外，传统的触觉反馈多为生硬的系统提示，未能与环境和多模态信息深度融合。新一代交互要求触觉能够对**声音、视频、动作、文字、表情、情绪、空间、位置**等全方位多模态要素做出精准、细腻的触觉级实时反馈，这迫切需要一套高内聚、低耦合的标准协议。

## 1.2 当前行业痛点

| 痛点 | 说明 |
| :--- | :--- |
| **硬件碎片化** | 不同厂商的马达（如 ERM、LRA、压电马达）特性差异巨大，同样的震动参数在不同手机上效果天差地别 |
| **语义缺失** | 现有 API 只提供“频率、振幅、时长”等底层参数，没有“心跳”、“上升”、“恐惧传递”等高阶概念 |
| **内容不可复用** | 一段精心设计的触觉脚本无法从一个 App 迁移到另一个，更无法跨平台、跨设备终端复用 |
| **创作门槛高** | 触觉设计需要专业波形工具和硬件调试，普通内容创作者、UI设计师或文本创作者无法参与 |
| **模态融合断层** | 触觉与声音、视频、空间物理位置及人类情感表情之间缺乏统一的映射逻辑与同步标准 |
| **高昂的标注成本**| 依赖人工逐帧为多媒体、文本、表情标注触觉或声光轨，难以应用到海量 UGC 短视频及流媒体实时反馈场景中 |
| **精细触觉与连续空间感缺失** | 【**v0.2.1 修订新增**】现有标准难以描述“蚂蚁爬过皮肤的细微痒感”或“指尖在屏幕上缓慢游走的平滑轨迹”，导致触觉生硬，缺乏自然连续性与细腻质感 |

## 1.3 本规范的目标与适用范围

**Multimodal Feedback Schema (MFS)** 是一套开放、免费、硬件无关的多模态触觉描述标准与全自动产品化解决方案。它定义：

- 一套基础多模态反馈词汇（从“轻触”到“空间环绕”，涵盖力量、节奏、空间、位置、运动、情绪、声光联动等维度）
- 一种通用脚本格式（HSF，类似 MIDI 或 SRT，支持多轨动态扩展与空间位置分发）
- 一套全模态内容（文本、视音频、表情、空间物理状态）到触觉与声光反馈的映射指南
- **一套完整产品化的四层全自动执行架构体系**，免除任何人工标注，实现端到端、流媒体级的内容特征实时提取、语义映射及硬件多点触觉渲染。

**适用范围**：移动设备、游戏手柄、智能穿戴设备（手套、背心）、VR/AR/MR 控制器、汽车智能座舱触觉提示、无障碍辅助设备等一切具备震动反馈、屏幕边缘渲染与外设 LED 闪光联动能力的终端。

# 1.A 新旧标准名称对比说明 (Name Transition Reference)

为了顺应交互技术从“单一触觉感受”向“声音、视频、动作、文字、表情、情绪、空间、位置全模态综合反馈”的演进，本白皮书自 v0.2 版本起，正式将项目名称由 `Haptic Event Schema (HES)` 升级为 `Multimodal Feedback Schema (MFS)`。新旧规范在框架、底层协议及核心逻辑上保持完全向下兼容，具体对应关系如下：

| 特性维度 | 原标准 (Legacy Standard) | 新标准 (Current Standard) | 演进核心方向 |
| :--- | :--- | :--- | :--- |
| **项目完整名称** | Haptic Event Schema | **Multimodal Feedback Schema** | 从“触觉事件”向“多模态综合反馈”泛化 |
| **技术简称** | HES | **MFS** | 契合多维感官融合生态 |
| **核心脚本格式** | HSF (Haptic Script Format) | **HSF (Haptic Script Format)** | 沿用 HSF 简称，但在结构中注入光影轨、触点坐标轨 |
| **覆盖通道域** | 仅覆盖马达单轴/多轴物理震动波形 | **马达多点空间对齐 + 屏幕边缘彩带 + 闪光灯 LED 脉冲** | 视、听、触、光多维物理线索协同调度 |
| **执行模式** | 依赖 HSF 脚本文件的静态解析 | **全自动无标注流式解析 + 静态 HSF 双模驱动** | 引入实时多模态 AI 提取层，具备全自动泛化能力 |

# 2. 设计原则

## 2.1 硬件无关性 (Hardware Agnostic)

MFS 只描述“应该感受到什么与呈现何种光影”（What to feel & see），而不规定“具体怎么执行波形驱动”。具体波形渲染由各平台的实现层（Renderer）根据本地马达硬件特性（如共振频率、响应时间）及 LED 控制器进行动态最优化适配与渲染。

## 2.2 语义优先于波形 (Semantic over Waveform)

开发者优先使用高级语义标签 `Rhythm.Heartbeat`、`Space.EdgeOrbit` 或 `Emotion.Tension`，而非底层硬件控制参数。语义层保证了跨设备体验的主观一致性，使触觉和多模态反馈具备了可读性与编辑性。

## 2.3 组合性与扩展性 (Composable & Extensible)

基础词条可以组合成复合反馈（如“空间接近 + 边缘彩带转动 + 闪光灯闪烁”）。同时允许开发者自定义扩展词条（保留私有编码区），并支持通过参数化包络（Envelope）对基础词条进行二次微调。

## 2.4 向后兼容 (Backward Compatibility)

新版本词典必须保持旧编码的有效性。不支持高级空间感、屏幕边缘彩带或闪光灯控制的低端硬件设备应具备智能降级策略（如自动忽略特定轨、自动替换为基础的 `Tap.Light` 或选择性静默）。

## 2.5 多模态对齐性 (Multimodal Alignment)

触觉与物理反馈事件必须能够与声音的时域/频域特征、视频的光流/色彩突变、文字的语义情感倾向、表情与情绪的强弱变化，以及空间三维位置的变化保持微秒级的精准同步与线性/非线性映射。

## 2.6 全自动免标注解析 (Zero-Labeling Automated Analysis)

系统在全面产品化的演进中引入了多模态人工智能中枢，跳出了单一依靠“静态预标注文件”的局限，能够像人类大脑一样，实时在端侧或云端自动解析输入内容中的音视频与文本表情高能特征，实现全自动无标注的物理触觉即时渲染。

## 2.7 零等待即时流式体验 (Zero-Wait Streaming Experience) 【**v0.2.1 修密新增**】

系统在产品化设计上必须达到“即开即感”的消费级流畅度。用户无需等待内容加载、无需手动选择“开启多模态”模式、无需任何预标注或预处理过程。从手指滑动短视频的第一帧起，多模态反馈应如影随形，绝对消除“正在生成触觉反馈...”的等待感与割裂感，使多模态感受成为内容的天然属性，而非附加功能。

# 3. 基础触觉与多模态分类体系 (Taxonomy)

我们将人类可感知的触觉、光影微反应以及多模态触发源（声音、视频、动作、文字、表情、情绪、空间、位置）归纳并映射为 **5 大维度**，共 **18 个子类**。

## 3.1 优先级、驱动原则与多点声光联动逻辑

1. **多点分驱与轨迹传导原则**：本协议完美支持现代多线性马达、阵列式压电马达及可穿戴触点。系统可实现**手机左下（Bottom-Left）、右下（Bottom-Right）、左上（Top-Left）、右上（Top-Right）**四大核心物理象限的独立或组合式震动。在此基础上，支持两维平面的四向定向位移传导：**从上到下（Top to Bottom）、从左到右（Left to Right）、从右到左（Right to Left）、从下到上（Bottom to Top）**，以及围绕屏幕/设备物理边界的**线性持续环绕震动（Linear Continuous Orbital Vibration）**，以提供高保真的空间运动位移感。
2. **屏幕边缘彩带联动原则（Screen Border Ribbon Orbiting）**：当系统触发特定空间或运动反馈时，屏幕边缘应同步渲染彩带环绕转动效果。该动态视觉由**速度等级（Speed Levels 1–4）**与**运动方向（顺时针 Clockwise / 逆时针 Counter-Clockwise）**二维控制，用于强化空间位移的视觉停留感。
3. **闪光灯情绪联动原则（Flashlight LED Strobing）**：手机后置闪光灯/前置补光灯被纳入多模态情感反馈链。系统根据底层情感倾向检测模型或表情识别所得的**情绪严重程度/强弱系数（Emotion Amplitude 0.0–1.0）**，动态映射闪光灯的**开启状态、闪烁频率（Strobing Frequency 1Hz–20Hz）与亮度脉宽百分比**，实现生理情绪状态的外显化物理级反馈。
4. **一维与多维动态混合原则**：同一轨道在同一微秒级时间切片内默认遵循**“强度取大（Max Amplitude Principle）+ 频率混叠（Frequency Mixing）”**原则。高优先级模态事件（如用户实时 UI 交互点击）在硬件执行层具有高仲裁权，可实时动态修正或叠加在低优先级的背景氛围多模态轨之上。
5. **严格基于内容映射**：所有反馈必须严格基于多模态输入源特征。文字内的愤怒情感高分值必须触发压迫反馈与高频闪光，音频里的重低音必须触发重击，禁止乱序脱节。
6. **五维优先级铁律**：**情绪感(Emotion) > 运动感(Motion) > 力量感(Force) > 空间与位置感(Space) > 节奏感(Rhythm)**。高优先级事件允许在硬件层随时打断或重构低优先级事件。
7. **连续空间游走与精细纹理映射原则（Continuous Spatial Trajectory & Micro-texture Rendering）** 【**v0.2.1 修订新增**】：为满足“游走”与“蚁行感”的自然体验，系统引入 **亚象限连续强度插值（Sub-quadrant Continuous Intensity Interpolation）** 与 **微秒级纹理脉冲簇（Micro-Texture Pulse Cluster）** 技术。空间轨迹不再局限于四象限的离散跳跃，而是通过相邻马达的实时动态功率配比，在物理边界内模拟出任意角度、任意速度的平滑移动感；同时，针对瘙痒、轻抚等微弱触感，支持 5-20ms 间隔的超低幅（强度<0.2）非对称随机脉冲串，以模拟生物体的蠕动感。

# 3.B 产品化四层闭环执行架构 (Four-Layer Productized Architecture)

为了实现从**海量未知多媒体/实时流媒体内容**到**物理多模态终端反馈**的全自动、零标注转化，MFS 在产品化实现层落地了完整的端到端四层技术架构。这一架构在系统内部实现毫秒级流水线闭环：


```
┌────────────────────────────────────────────────────────┐
│               1. 原始内容多模态输入层                    │
│     (海量UGC短视频、实时流媒体、即时文本、面部摄像头等)       │
└───────────────────────────┬────────────────────────────┘
│
▼
┌────────────────────────────────────────────────────────┐
│           2. 多模态情绪识别 AI 层 (端侧/云端)           │
│   (VAD/LLM文本情感 / CV光流与面部表情FACS / 音频时频提取)    │
└───────────────────────────┬────────────────────────────┘
│
▼
┌────────────────────────────────────────────────────────┐
│                  3. MFS 标准映射层                     │
│  (全自动规则引擎与高维向量对齐，将多维特征即时转化为MFS词条)   │
└───────────────────────────┬────────────────────────────┘
│
▼
┌────────────────────────────────────────────────────────┐
│                 4. MFS 实时渲染引擎                    │
│ (四象限原生马达总线 / iOS CoreHaptics / 闪光灯与彩带Canvas)  │
└────────────────────────────────────────────────────────┘
```

1. **原始内容多模态输入层 (Input Layer)**：直接承载海量 UGC 短视频、动态流媒体、用户交互行为、即时社交文本以及前置摄像头采集到的实时面部动作。该层无需任何预先编写的 `*.hsf` 脚本文件，直接将原始多媒体数据流注入流水线。
2. **多模态情绪识别 AI 层 (Multi-Modal AI Inference Layer)**：内嵌轻量化端侧大模型与高性能 CV/Audio 推理算子。
   - **文本域**：通过小型化 LLM 或双向 Transformer 实时提取语意、语气助词及隐含的情感倾向分值；
   - **视觉域**：利用高效光流场算法（Optical Flow）提取视频运动轨迹与镜头推拉运动，并采用面部动作编码系统（FACS）提取微表情特征点；
   - **音频域**：通过短时傅里叶变换（STFT）提取瞬态时域能量、频谱质心与重低音包络。
3. **MFS 标准映射层 (MFS Mapping Layer)**：将 AI 层输出的高维多模态特征矩阵，通过基于“优先级铁律”的确定性规则引擎与向量空间对齐算法，**全自动无标注地在线重构为 MFS 标准词条序列**，在内存中动态生成瞬时多轨事件流。
4. **MFS 实时渲染引擎 (MFS Rendering Engine)**：最终的底层硬件消费级驱动。解析瞬时事件流，对设备四象限物理马达进行独立分流波形合成，控制屏幕边缘 Canvas 渲染彩带，并通过高精度定时器发射特定 PWM 脉冲以驱动后置 LED 闪光灯。

# 4. 力量感 (Force) —— 震动的“重量”与“锐度”

常用于映射物理碰撞、动作打击、视频中的爆炸、文字中的强语气表达。

| 子类 | 英文 | 描述 |
| :--- | :--- | :--- |
| 轻触 | Tap | 蜻蜓点水，极短促，常用于文字按键、UI微交互 |
| 敲击 | Knock | 有明确起止的撞击感，对应普通步态、常规鼓点 |
| 重击 | Heavy | 沉闷、有力、下坠感，对应重物落地、重拳动作 |
| 爆发 | Burst | 瞬间释放的巨大能量，对应视频爆炸、闪光、情绪破防 |
| 震荡 | Oscillate | 持续高频的颤抖，对应引擎轰鸣、声乐长音、持续警报 |
| 碾压 | Crush | 逐渐加重直至极限，对应骨骼碎裂、机械挤压、极度压抑文本 |
| **蠕动 / 蚁行** | **Crawl / Itch** | 【**v0.2.1 修订新增**】极度轻微（强度≤0.2）、间隔不规则（5-20ms）的短促脉冲序列，模拟蚂蚁爬行、羽毛轻抚、皮肤瘙痒等精细微触感。 |

---

# 5. 节奏感 (Rhythm) —— 震动的“时间排列”

常用于映射音频BPM、呼吸节奏、动作频次、心率变化。

| 子类 | 英文 | 描述 |
| :--- | :--- | :--- |
| 单次 | Single | 孤立的、一次性的触觉独立单元 |
| 重复 | Repeat | 均匀的来回往复，如钟表滴答、规则机械运转 |
| 脉冲 | Pulse | 规律且带有间隙的推进，对应雷达扫描、警示闪烁 |
| 心跳 | Heartbeat | 强弱交替的“咚哒”模式，直接映射生物心率与紧张表情 |
| 波浪 | Wave | 强度呈正弦曲线起伏，对应呼吸感、水流涌动、抒情音乐 |
| 滚奏 | Roll | 极快速密集的连续敲击，对应机枪射击、鼓点SOLO、动作疾跑 |

---

# 6. 空间感与位置感 (Space & Location) —— 震动的“方位、距离与多点轨迹”

用于对空间三维物理位置、绝对/相对坐标、视频运动镜头、三维音频以及多触点方向位移做出位置级声光震反馈。

| 子类 | 英文 | 描述 |
| :--- | :--- | :--- |
| 接近 | Approach | 由弱变强。映射物体或光源靠近、距离坐标缩短，触发边缘彩带汇聚 |
| 远离 | Retreat | 由强变弱。映射物体离去、距离坐标拉远，边缘彩带消散 |
| 穿过 | Through | 四向传导（上→下、左→右、右→左、下→上），映射跨象限马达顺序传导 |
| 环绕 | Orbit | 物理四象限（左下、右下、右上、左上）执行线性持续环绕震动，彩带同步边缘转动 |
| 扩散 | Expand | 从中心向四周散开，映射波纹涟漪、视野张开、闪光灯瞬间闪烁 |
| 聚集 | Contract | 从四周向中心收拢，映射黑洞吸入、视线聚焦、位置坍缩 |
| **游走 / 轨迹** | **Trail / Glide** | 【**v0.2.1 修订新增**】通过相邻线性马达的连续动态功率配比（Cross-fade），实现目标触感在屏幕/设备二维平面内沿任意连续曲线（贝塞尔路径或光流轨迹）平滑移动，消除触点跳跃感，模拟指尖滑移或物体漂移。 |

> *注：复杂的空间感、位置感与多触点传导需要多马达矩阵或智能穿戴设备支持，单马达设备上将降级为宏观的全局振幅渐变。*

# 7. 运动感 (Motion) —— 震动的“物理惯性与动作特征”

用于精准对齐物理动作行为、视频运镜速度（如推拉摇移）、动态手势及文字语速变化。

| 子类 | 英文 | 描述 |
| :--- | :--- | :--- |
| 上升 | Rise | 频率或强度递增，对应跳跃起飞、情绪激昂、音调拉高 |
| 下降 | Fall | 频率或强度递减，对应失重坠落、画面俯冲、音调沉降 |
| 加速 | Accel | 间隔越来越短，对应奔跑加速、心率飙升、动作节奏紧凑、彩带环绕速度等级提升 |
| 减速 | Decel | 间隔越来越长，对应汽车刹车缓冲、动作收尾、语速放缓、彩带速度级下降 |
| 悬浮 | Hover | 极度轻微且持续的飘忽感，对应零重力状态、微风拂面、平静表情 |
| 急停 | Brake | 高速震动瞬间归零，对应动作硬直、突发碰撞、情绪顿悟、闪光灯骤灭 |

# 8. 情绪感与表情感 (Emotion & Expression) —— 震动的“心理暗示与生理多模态微反应”

用于映射文本情感色彩、面部表情微表情编码（FACS）、视频氛围及音频调性，并强制绑定闪光灯闪烁频率。

| 子类 | 英文 | 描述 |
| :--- | :--- | :--- |
| 紧张 | Tension | 高频细碎、不规律，对应焦虑表情，闪光灯高频低亮不规则闪烁 |
| 压迫 | Oppress | 沉重、喘不过气，对应愤怒面容、巨兽凝视，闪光灯低频超高亮度输出 |
| 释放 | Release | 紧张后的突然松弛，对应叹气、大笑、危机解除，闪光灯平滑淡出 |
| 兴奋 | Excite | 高频率、高强度的连续刺激，对应狂喜、爆点，闪光灯达 20Hz 极限频率满载爆闪 |
| 平静 | Calm | 极低频、极柔和的包裹感，对应微笑表情、舒缓文本，闪光灯长亮或柔和呼吸灯律动 |
| 诡异 | Weird | 不和谐的非对称、非线性震动，对应嫌恶/阴险表情，闪光灯非周期性突发闪烁 |

# 9. 事件编码规范

## 9.1 编码规则

每个标准事件由 **前缀字母 + 三位数字** 组成，前缀代表所属维度：

| 前缀 | 维度 | 示例 | 映射主要多模态数据源 |
| :--- | :--- | :--- | :--- |
| F | 力量感 (Force) | F-101 | 声音振幅、物理碰撞、视频撞击、动作力度 |
| R | 节奏感 (Rhythm) | R-104 | 音频BPM、心率变化、动作频次、语言节奏 |
| S | 空间感与位置感 (Space) | S-201 | 3D空间坐标、多触点物理象限、视频运镜、屏幕彩带、相位 |
| M | 运动感 (Motion) | M-301 | 动作速度矢量、视频光流速度等级、物理惯性、语速 |
| E | 情绪感与表情感 (Emotion)| E-401 | 文本情感得分、面部表情特征点、闪光灯 LED 闪烁频率 |

数字段从 001 开始。为了保证扩展性，规范规定：**000–099 为私有扩展/自定义区**（保留给各厂商开发者自由定义特殊硬件波形与私有声光交互）；**100–999 为公开标准词条**（遵循 MFS 标准规范）。

## 9.2 事件属性

每个词条在 Schema 定义中应至少包含以下属性：

| 属性 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | string | 唯一编码，标准词条必须以公开标准区间（100-999）命名，如 `F-101` |
| `name` | string | 标准英文名，采用点分法表示层级，如 `Tap.Light` |
| `cn_name` | string | 中文名，如 `轻触` |
| `category` | string | 所属维度，取值为 `Force`, `Rhythm`, `Space`, `Motion`, `Emotion` |
| `intensity_default` | float (0–1) | 默认强度基准值 |
| `duration_ms` | int | 建议持续时间（毫秒），-1 表示由关联源动态决定持续时间 |
| `waveform_hint` | string | 波形几何及包络建议（方波/正弦波/锯齿波/白噪波/自定义包络） |
| `multimodal_link` | object | 多模态联动控制块（包含局部触点马达分配、彩带速度与方向、闪光灯频率） |
| `description` | string | 物理及生理心理感受的详细文本描述 |

# 11. 触觉脚本格式 (HSF) 规范

## 11.1 文件结构

HSF 使用 JSON 作为载体，具备高阶多轨扩展性。在 MFS 体系下，可对设备不同物理象限的马达（如 `motor_top_left`）进行高精度独立分轨，同时无缝挂载屏幕彩带轨（`screen_ribbon_track`）与外设闪光灯轨（`flashlight_led_track`）。【**v0.2.1 修订新增**】为支持连续游走（`S-216`），新增 `trail_path` 参数块，用于定义二维贝塞尔路径或坐标点数组。

## 11.2 字段定义
| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| schema | string | 是 | 固定为 "HapticScriptFormat" |
| version | string | 是 | 版本号，当前演进版本为 "0.2.2" |
| meta | object | 否 | 元数据，包含标题、总时长、BPM及全维多模态参考源信息 |
| tracks | array | 是 | 轨道列表，各轨道物理绑定一个具体的控制通道或感官呈现媒介 |
| track.id | string | 是 | 轨道唯一标识符 |
| track.device_target | string | 否 | 设备类型提示（如 phone_multi_motor, phone_screen_edge, rear_flash_led） |
| track.location_tag | string | 否 | 物理空间位置标签（如 hand_left, screen_border, device_back） |
| track.events | array | 是 | 事件列表，必须在轨道内按照时间戳升序严格排列 |

| event.duration_ms | integer | 否 | 持续时长。若缺省则默认调用标准事件词典中的 duration_ms |
| event.event_ref | string | 是 | 标准事件词典编码（如 S-215, E-407 等） |
| event.intensity | float | 否 | 强度缩放系数（0.0–1.0），用于对词典默认物理强度进行增益调节 |
| event.ribbon_parameters | object | 否 | v0.2新增：屏幕边缘彩带专用控制块，定义 direction 与 speed_level |
| event.led_parameters | object | 否 | v0.2新增：外设闪光灯专用控制块，定义 emotion_severity 与计算频率 |
| event.trail_parameters | object | 否 | 【**v0.2.1 修订新增**】连续游走专用控制块，包含 path_points（二维坐标数组）、speed（速度系数）、loop（是否循环） |
| event.note | string | 否 | 供编辑、标注使用的描述性备注（如多模态对齐提示） |
| event.envelope | object | 否 | 参数化自定义包络，用于重写或微调默认物理波形特性 |
## 11.3 排序与合并法则
 * **严格时间序**：所有轨道内部的事件对象必须严格按照 time_ms 进行升序排列，以便于流式渲染引擎高效读取和降低寻址开销。
 * **重叠混合策略（Mixing Rule）**：当同一物理轨道在同一时间段内出现两个或多个 MFS 事件重叠时，系统渲染器默认遵循本白皮书第 3.1 节确立的“五维优先级铁律”进行硬件独占性覆盖或频率重构混合，禁止产生手感浑浊、视觉撕裂或光影重叠。
 * **硬件保护限流与防削波**：事件触发频率建议在单触点硬件上不超过 8 次/秒。高密度的多点环绕和极限频闪（E-404 20Hz）在渲染层需受限于本地系统过热追踪保护模型。

# 13. 开发者与全自动系统产品化规范
## 13.1 全自动 AI 推理内核时序规范
全自动无标注流式解析要求多模态情感识别 AI 层与映射层必须在设备端侧以极低开销常驻运行。其核心处理时序受到硬件级硬限制：
 1. **滑动分析窗 (Sliding Window)**：视频流、音频流与文本轴的实时分析窗宽度限制在 30ms - 100ms 之间，过大将导致用户可察觉的物理延迟，过小将引发高能特征捕获漏判。
 2. **多源多线程数据对齐**：视频帧与音频PCM信号由底层的统一硬件基准时钟（Host Master Clock）进行微秒级标记，防止 AI 层推理时间差引发声、光、震错位。
## 13.2 解析器（Parser）与映射内核要求
 * **完备的 JSON 校验**：解析器必须能够支持标准 JSON 格式的 HSF 文本流或二进制流的解析，对非法字段、嵌套错误及 v0.2 新增的彩带、LED 扩充块及 v0.2.1 新增的 trail_parameters 块需具备完备的容错捕获机制。
 * **动态映射与无标注泛化**：当解析器运行在全自动实时流模式下，映射内核必须能够直接承载来自 AI 推理层实时吐出的连续高维情感与物理参数特征（如情绪强弱度、光流速度标量及二维轨迹向量），在无需外部 HSF 脚本参与的前提下，瞬时计算并产生与之完全线性/非线性对应的多轴马达、Canvas及闪光灯控制流。
 * **版本向前兼容与识别**：必须严格读取并解析 version 字段。若遇到高版本 HSF 文件，当前 v0.2.2 解析器必须能够忽略不支持的未来参数，并正确提取基础事件字段，同时能完美向下兼容 v0.1 旧脚本文件。
 * **未知事件防崩溃 (Fallback Map)**：对于任何自定义或当前版本不支持的 event_ref（如无法识别的细分光影编码），解析器不得报错中断，必须将其统一映射到通用降级表（详见 13.5）。
## 13.3 渲染器最小实现（Android / iOS）
各平台渲染实现层负责将抽象的 MFS 事件实时翻译为系统原生的触觉、图形层边缘动画与 LED 闪光灯 API。
| 平台 | 基础多模态映射逻辑 | 降级策略实现 |
|---|---|---|
| **Android** | 将标准 event_ref 或动态映射层生成的物理参数转换为原生的 android.os.VibrationEffect 数组（利用多线性马达支持）；彩带图层通过挂载在 Window 边缘的自定义 Canvas View 执行渲染；闪光灯调用 CameraManager.setTorchMode() 实现特定 PWM 频率控制。针对 S-216，采用分时动态调整各马达振幅（VibrationEffect 组合）实现交叉淡入淡出。 | 当设备不支持振幅控制、多点分驱或不具备闪光灯时，彩带及闪光灯块直接静默忽略，所有不支持事件一律退化为全局单次震动：VibrationEffect.createOneShot(50, VibrationEffect.DEFAULT_AMPLITUDE) |
| **iOS** | 利用 CoreHaptics 框架将事件参数构造为多点 CHHapticPattern 字典并执行播放；彩带通过 CoreAnimation 在当前 UIWindow 四周绘制边缘不相交渐变层；闪光灯通过 AVCaptureDevice 的 torchLevel 配合高精度定时器执行频率频闪控制。针对 S-216，利用 CHHapticParameterCurve 实现多马达强度之间的平滑动态插值。 | 当运行在旧款 iOS 设备时，光影效果及高级环绕自动静默降级，退化为传统的系统硬震反馈：UIImpactFeedbackGenerator(style: .medium).impactOccurred() |
## 13.4 性能与时延要求
 * **全自动无标注端到端总时延**：从多媒体帧到达输入层，经由「情绪识别AI层 → 标准映射层 → 渲染引擎」完成物理动作爆发，端侧全链路硬件总时延必须控制在 ≤ 12ms，以保障视觉、听觉与多触点反馈物理完全实时同步。
 * **零等待即时启动（Zero-Wait TTFF）** 【**v0.2.1 修订新增**】：从用户滑动切换至新内容的瞬间（首帧解码完成）到渲染引擎输出第一组多模态物理反馈（首感），时延必须 ≤ 50ms。系统严禁显示“加载中”、“生成反馈中”等阻断式交互，确保用户体验如同刷原生抖音般绝对流畅、无感衔接。
 * **动态解析开销**：单次解析 HSF 文件或段落的时间必须 ≤ 5ms（针对 10 分钟或 2000 个事件长度的脚本）。
 * **多模态实时调度时延**：从接收到事件到调用系统 API 发出震动指令、边缘彩带起舞、闪光灯亮起的三端异构软件调度延迟差必须小于 5ms，整体时延 < 10ms。
 * **CPU 实时全自动推理占用**：多模态情绪识别 AI 层在端侧持续运行时，多感官核心调度线程与轻量 AI 推理内核的 CPU 总占用率必须 < 4.5%（基于主流旗舰级移动端 SoC 测试平台）。
 * **内存开销**：常驻内存缓冲区的静态词典、轻量级模型权重与动态事件队列总开销必须 < 45MB。
## 13.5 智能降级策略（Fallback Strategy）
 * **硬件基础能力不支持**：若目标终端设备仅搭载单轴马达且无外设闪光灯，渲染层应自动对脚本实施脱水剥离，剔除彩带及 LED 属性，并将所有复杂的空间、运动事件退化为标准的 F-101 (Tap.Light)。针对 F-107 (Crawl) 与 S-216 (Trail) 将统一退回至 F-101。
 * **CPU 系统过载保护与 AI 内核自动降级**：当检测到终端系统整体 CPU 负载过高或系统步入过热红区时，产品化架构将主动对“多模态情绪识别 AI 层”实施熔断性功能降级——关闭视频光流（含轨迹追踪）与复杂的面部 FACS 追踪，AI 推理退化为仅依靠超轻量级音频时频能量段进行低算力级触觉反馈，保障系统整体界面 UI 的绝对流畅度。
 * **马达与外设 LED 物理过热保护**：由于长时间 20Hz 满载暴闪与强震会导致线圈和 LED 芯片发热严重，渲染层应建立虚拟温度追踪模型，当累计满载时间触发限额阈值时，自动等比例调低闪光灯亮度与马达 intensity 系数（如降低 30% 功率输出），进入硬件安全保护状态。
# 14. 应用场景
## 14.1 短视频与音乐流媒体（捉秀 App 全自动智能化核心载体）
通过「多模态情绪识别AI + MFS映射层 + MFS渲染引擎」的完整产品化四层架构，用户在使用捉秀 App 刷任何突发的 UGC 短视频、看直播或收听海量音乐时，系统可在完全无需人工标注、无需预配置 HSF 脚本、无需用户手动开启任何“多模态模式”的理想状态下，全自动实时在线解析视频流与音浪节奏。
【**v0.2.1 修订新增**】 这不仅意味着手机能随画面爆炸或鼓点重击而震动，更进化到了可感知内容的“细微皮肤触感”与“平滑空间位移”——当你刷到一条羽毛轻抚镜头或昆虫爬行的特写视频，手机会自动输出如蚂蚁在掌心游走的微弱瘙痒感（F-107）；当画面中的热气球缓慢从左上方漂移至右下方时，四颗马达会像接力赛般平滑传递震感，让您闭眼也能感知物体的精确漂移轨迹（S-216）。
整个过程中，从手指滑动屏幕、视频起播到多模态感受输出，无缓冲等待、无感知延迟（TTFF ≤ 50ms），实现了真正的“视、听、触、光”全沉浸无感化体验。
## 14.2 无障碍辅助感知
为视障或听障人群提供开创性的多模态辅助导航与信息感知。将周围环境的物理障碍距离（空间接近 S-201 / 远离 S-202）、路线转弯导航指示、周围人群的面部情绪和表情特征等，实时翻译为特定的编码触觉，通过可穿戴触觉设备传递给视听障用户。
## 14.3 游戏互动深度反馈
将游戏内复杂多变的环境事件解耦。主角的脚步声动作、不同枪械的后坐力力量感、远处的突发爆炸、敌人的空间接近方位坐标，皆可通过 HSF 轨道脚本无缝映射至手柄或震动背心中，大幅提升核心玩家的游戏临场感。
## 14.4 智能座舱与驾驶安全预警
在汽车智能座舱中，当车辆遭遇车道偏离、盲区障碍物快速接近、前车急刹等紧急物理状况时，系统不再单靠刺耳的警报声，而是通过方向盘、座椅的多点多马达矩阵输出高强度的空间运动触觉（如 M-306 急停 + E-401 紧张），实现比听觉更直接、人类生理反应更迅速的安全预警。
## 14.5 通知信息语义化（无感通知）
改变千篇一律的系统默认震动。不同的社交软件、不同的人物联系人（如紧急联系人与普通群聊）、不同的通知语义类型（微信重要提及、日历闹钟、邮件送达），均可被分配专属且具备明显心理暗示的 MFS 事件词条。用户无需点亮和观看屏幕，仅凭身体接触即可瞬间获知消息的重要程度。

# 附录 C：测试用例与标准测试曲目集
本曲目集用于检验不同硬件终端执行 MFS 多模态解析与多点声光震渲染时的还原度与主观表现力。
| 标准曲目/测试用例场景 | 预期触发的核心 MFS 标准事件 | 评测合格基准线说明 |
|---|---|---|
| **全景声大片动作运镜场景** | 镜头由左至右横向划过 → S-212，紧接着发生大爆炸 → S-205 + F-104 | 物理震动与屏幕彩带需完美体现从左侧向右侧滑动的位移轨迹感；爆炸瞬间马达重击、彩带全屏炸裂与闪光灯暴闪三者软件调度时差不得超过 5ms。 |
| **电子乐（如《Animals》）** | 极低频 Drop 连续能量段 → E-402 (Oppress)，副歌高潮极限飙升 → E-404 | Drop 段落要求马达稳定输出 40Hz 重压，后置闪光灯以 2Hz 同步维持高亮频闪；高潮阶段马达、全屏彩带与闪光灯必须全面满载，触发 20Hz 极限暴闪，且硬件不应严重过热。 |

| **精细纹理与连续游走测试（特写及空镜）** | 【**v0.2.1 修订新增**】 视频内容为羽毛轻抚手臂（缓慢斜线位移）+ 字幕提及“痒”。 | 必须同时触发 F-107 (Crawl) 输出微弱不规则脉冲，且 S-216 (Trail) 驱动四马达平滑接力转移震感，中间不得出现断层或跳跃。评测员闭眼需能感知出位移轨迹为“左上→右下”平滑斜线。 |
| **抖音级零等待体验测试（盲测）** | 【**v0.2.1 修订新增**】 随机滑动 50 个不同类型 UGC 短视频。 | 用户感知不到任何“加载/生成反馈”的卡顿或延迟，主观评价“就像视频自带震动一样自然”。系统日志记录 TTFF（首感时延）最大峰值必须 ≤ 50ms。 |
# 附录 D：术语表 (Glossary)
| 术语名称 | 全称 / 英文原词 | 行业标准定义与本白皮书规范上下文解释 |
|---|---|---|
| **MFS** | Multimodal Feedback Schema | **多模态反馈模式**。本白皮书所定义的，面向声音、视频、动作、文字、表情、情绪、空间、位置等全维度多模态交互的基础分类体系、编码规范以及多感官语义字典的总称。 |
| **HSF** | Haptic Script Format | **触觉与多模态脚本格式**。基于 JSON 结构设计的多轨制综合反馈描述文件规范，作为承载 MFS 物理事件的核心文本载体。 |
| **线性持续环绕震动** | Linear Continuous Orbital Vibration | 通过精细控制设备空间内部不相交物理象限马达的起振时刻与包络相位差，使人在触觉层面产生震动信号正沿硬件边界平滑滚动环绕的物理心理错觉。 |
| **边缘彩带转动** | Screen Border Ribbon Orbiting | 一种围绕屏幕物理边缘渲染的动态线状视觉特效，通过传入的速度等级（1-4级）和运动方向，与空间/运动物理马达轨迹保持实时同频。 |
| **闪光灯情绪频闪** | Flashlight LED Strobing | 依据上层情感倾向或面部微表情识别所得的强弱系数，将后置/前置 LED 闪光灯转化为特定脉冲频率（1Hz-20Hz）的生理情绪外显物理光影反馈。 |
| **连续游走（轨迹）** | Continuous Trail / Glide | 【**v0.2.1 修订新增**】 通过相邻多颗线性马达的实时动态功率交叉淡变（Cross-fade），使触觉聚焦点能够在二维平面内沿任意平滑路径连续移动，模拟物理物体滑移或生物蠕动轨迹，杜绝离散跳跃感。 |
| **蚁行感（纹理）** | Crawl / Itch Texture | 【**v0.2.1 修订新增**】 一种超低幅、短促、时域分布非均匀的脉冲串触觉，专门用于映射“蚂蚁爬行”、“羽毛轻抚”等皮肤级微痒感知，要求极低的强度（≤0.2）和伪随机的间隔（5-20ms）。 |
| **渲染器 (Renderer)** | Multimodal Renderer | 驻留在系统底层或 SDK 内部的运行渲染模块，负责解析 HSF 文件，并将抽象的 MFS 语义编码高效转化为具体平台硬件直接执行的波形控制数据流、 Canvas 动画以及外设驱动信号。 |
